01. UNIVERSAL SCCM TROUBLESHOOTING FLOW
============================================================

Before troubleshooting any SCCM problem, follow this order:

  1. Confirm the device is discovered in SCCM.
  2. Confirm SCCM client is installed.
  3. Confirm client is Active.
  4. Confirm device is in the correct collection.
  5. Confirm the deployment is enabled.
  6. Confirm deployment available time has already passed.
  7. Confirm deployment is visible in Software Center if needed.
  8. Confirm client received policy.
  9. Confirm detection / scan / evaluation result.
  10. Confirm content is distributed to DP.
  11. Confirm boundary group has correct DP / MP / SUP.
  12. Confirm download works.
  13. Confirm install command works manually.
  14. Confirm client sends state message back to server.
  15. Confirm console status updates after summarization.

Most SCCM problems follow this chain:

  DISCOVERY
      ↓
  CLIENT INSTALLATION
      ↓
  CLIENT COMMUNICATION
      ↓
  POLICY RECEIVED
      ↓
  EVALUATION / DETECTION
      ↓
  CONTENT LOCATION
      ↓
  DOWNLOAD
      ↓
  INSTALLATION
      ↓
  STATE MESSAGE / REPORTING

03. CLIENT INSTALLATION FAILED
============================================================

Symptoms:
  - SCCM client not installing
  - Client push fails
  - Device remains Client = No
  - ccmsetup.exe starts but stops
  - Software Center not installed

Most common root causes:
  - Client push account is not local administrator
  - Admin$ share disabled
  - Firewall blocks RPC/WMI/File and Printer Sharing
  - DNS name resolution problem
  - Wrong client installation command
  - Old/broken SCCM client remains
  - Antivirus blocks ccmsetup
  - Management Point not reachable
  - Boundary/site assignment problem

Important logs:
  Client:
    C:\Windows\ccmsetup\Logs\ccmsetup.log
    C:\Windows\ccmsetup\Logs\client.msi.log

  Site Server:
    ccm.log

Proven solution:
  1. From SCCM server, test DNS:
       ping CLIENTNAME
       nslookup CLIENTNAME

  2. Test admin share:
       \\CLIENTNAME\admin$

  3. Confirm client push account or SCCM server computer account is local admin on target device.

  4. Confirm firewall allows:
       File and Printer Sharing
       WMI
       RPC
       Remote Administration

  5. Check ccm.log on SCCM server.
       If access denied appears, fix admin permission.
       If network path not found appears, fix DNS/firewall/admin$.

  6. On client, open ccmsetup.log.
       Check download source, MP, and installation return code.

  7. Manual reinstall command:
       ccmsetup.exe /mp:SCCMSERVER FSP=SCCMSERVER SMSSITECODE=S01

  8. If client is corrupted, uninstall then reinstall:
       ccmsetup.exe /uninstall
       Wait until ccmsetup service is removed.
       Delete C:\Windows\ccmsetup if needed.
       Reinstall the client.

Verification:
  - Configuration Manager applet appears in Control Panel.
  - CcmExec service is running.
  - SCCM console shows Client = Yes.
  - Client Activity becomes Active.

Prevention:
  - Keep firewall GPO ready for SCCM clients.
  - Use correct client push account.
  - Make sure DNS and AD discovery are healthy.
  - Avoid cloning machines without sysprep.

04. CLIENT SHOWS "NO" IN CONSOLE
============================================================

Symptoms:
  - Device exists in SCCM console
  - Client column shows No
  - No Software Center on device

Most common root causes:
  - Device discovered from AD but client not installed
  - Client push failed
  - Client installed but not assigned to site
  - Duplicate/stale device record

Important logs:
  ccm.log
  ccmsetup.log
  ClientLocation.log
  LocationServices.log

Proven solution:
  1. Right-click device in console.
  2. Select Install Client.
  3. Check ccm.log on site server.
  4. Check ccmsetup.log on client.
  5. If manual install is needed, run:
       ccmsetup.exe /mp:SCCMSERVER SMSSITECODE=S01
  6. If duplicate record exists, delete old/stale record and allow rediscovery.
  7. Run Machine Policy Retrieval after installation.

Verification:
  - Device shows Client = Yes.
  - Client Check Result = Passed.
  - Policy Request time updates.

05. CLIENT INSTALLED BUT INACTIVE
============================================================

Symptoms:
  - Client = Yes
  - Client Activity = Inactive
  - Last policy request is old
  - Software Center does not update

Most common root causes:
  - SMS Agent Host service stopped
  - Client cannot reach Management Point
  - Boundary/site assignment issue
  - Certificate or registration issue
  - Firewall/DNS problem
  - Client health problem

Important logs:
  CcmExec.log
  ClientIDManagerStartup.log
  LocationServices.log
  CcmMessaging.log
  ClientLocation.log

Proven solution:
  1. On client, check service:
       services.msc
       SMS Agent Host = Running

  2. Restart service:
       net stop ccmexec
       net start ccmexec

  3. Confirm client site code:
       Control Panel > Configuration Manager > General tab

  4. Check LocationServices.log.
       Confirm MP is discovered.

  5. Check CcmMessaging.log.
       Confirm client can send/receive communication with MP.

  6. Run:
       Machine Policy Retrieval and Evaluation Cycle
       Discovery Data Collection Cycle

  7. If still inactive, reinstall client.

Verification:
  - Policy Request time updates.
  - Client Activity becomes Active.
  - CcmMessaging.log shows successful communication.

06. CLIENT NOT RECEIVING POLICY
============================================================

Symptoms:
  - Deployment does not appear
  - Software Center empty
  - Policy request time not changing
  - Client notification does not work

Most common root causes:
  - Client not in target collection
  - Machine policy not updated
  - MP communication issue
  - Client settings/policy corruption
  - Boundary issue
  - Wrong site code

Important logs:
  PolicyAgent.log
  PolicyEvaluator.log
  CcmMessaging.log
  LocationServices.log
  CCMNotificationAgent.log

Proven solution:
  1. Confirm device is in the deployment collection.

  2. Update collection membership:
       Right-click collection
       Update Membership

  3. Run Machine Policy Retrieval on client.

  4. Check PolicyAgent.log.
       Look for new policy download.

  5. Check PolicyEvaluator.log.
       Confirm policy evaluation is successful.

  6. Check CcmMessaging.log.
       Confirm MP communication is successful.

  7. If policy is stuck, restart SMS Agent Host.

  8. If still not fixed, repair/reinstall SCCM client.

Verification:
  - New deployment policy appears in PolicyAgent.log.
  - Software Center updates.
  - Deployment appears under correct tab.

07. SOFTWARE CENTER EMPTY
============================================================

Symptoms:
  - Software Center opens but no apps/updates
  - Applications tab is empty
  - Updates tab is empty
  - User says "nothing showing"

Most common root causes:
  - No deployment targeted to device/user
  - Client policy not received
  - Deployment is hidden
  - Available time has not arrived
  - Wrong collection
  - App requirements not met
  - User/device deployment confusion

Important logs:
  SCClient*.log
  SoftwareCenterSystemTasks.log
  PolicyAgent.log
  AppIntentEval.log
  UpdatesDeployment.log

Proven solution:
  1. Confirm deployment target:
       Device deployment -> device must be in collection.
       User deployment -> user must be in user collection.

  2. Check deployment purpose:
       Available deployments appear for users/devices.
       Required deployments may install automatically depending on settings.

  3. Check User Experience:
       Must not be hidden if user needs to see it.

  4. Check Scheduling:
       Available time must already pass.

  5. Run Machine Policy Retrieval.

  6. Restart Software Center or restart SMS Agent Host.

  7. Check logs:
       PolicyAgent.log for policy
       AppIntentEval.log for application
       UpdatesDeployment.log for updates

Verification:
  - Deployment appears in Software Center.
  - Logs show policy received and evaluated.

08. APPLICATION NOT SHOWING IN SOFTWARE CENTER
============================================================

Symptoms:
  - Application is deployed
  - Device is in collection
  - App not visible in Software Center

Most common root causes:
  - Deployment available time not reached
  - User Experience hidden
  - App deployed to user but user not logged in
  - Requirement rule not satisfied
  - Device not updated in collection membership
  - Client policy not refreshed
  - Application approval required
  - App is superseded/replaced

Important logs:
  PolicyAgent.log
  AppIntentEval.log
  AppDiscovery.log
  SCClient*.log

Proven solution:
  1. Check deployment collection membership.

  2. Update collection membership.

  3. Check deployment:
       Purpose = Available or Required
       Enabled = Yes
       Available time = Now/past
       User notification = Display in Software Center

  4. Run Machine Policy Retrieval.

  5. Check AppIntentEval.log.
       If requirements not met, fix requirement rule.

  6. If app is user-targeted, confirm correct user is logged in.

  7. If approval is required, approve application request.

Verification:
  - AppIntentEval.log shows app applicable.
  - Software Center displays the application.

09. APPLICATION DOWNLOAD STUCK
============================================================

Symptoms:
  - Application stuck at Downloading
  - Download stuck at 0%
  - Error about content not available
  - Software Center retry does not help

Most common root causes:
  - Content not distributed to DP
  - Boundary group missing/wrong
  - Client cannot find DP
  - DP is down/unhealthy
  - BITS issue
  - Client cache full
  - Network/proxy issue

Important logs:
  CAS.log
  ContentTransferManager.log
  DataTransferService.log
  LocationServices.log
  AppEnforce.log

Proven solution:
  1. Confirm content is distributed:
       Monitoring > Distribution Status > Content Status

  2. If failed, redistribute content to DP.

  3. Check boundary group:
       Client IP/subnet must belong to boundary.
       Boundary must be in boundary group.
       Boundary group must have DP assigned.

  4. Check CAS.log:
       Confirm client finds DP.

  5. Check ContentTransferManager.log:
       Confirm download job starts.

  6. Check DataTransferService.log:
       Confirm BITS download.

  7. Clear client cache if full:
       Control Panel > Configuration Manager > Cache tab
       or use client cache cleanup script.

  8. Restart SMS Agent Host and retry.

Verification:
  - CAS.log shows content location found.
  - DataTransferService.log shows download completed.
  - App installation starts.

10. APPLICATION INSTALLATION FAILED
============================================================

Symptoms:
  - Application appears but fails to install
  - Software Center shows Failed
  - Error code appears
  - Installer works manually but not via SCCM

Most common root causes:
  - Wrong install command
  - Silent switch missing
  - Installer requires user interaction
  - Runs as SYSTEM but needs user profile
  - Missing prerequisite
  - Wrong working directory
  - Reboot pending
  - Antivirus blocks install
  - Exit code not handled

Important logs:
  AppEnforce.log
  AppDiscovery.log
  AppIntentEval.log

Proven solution:
  1. Open AppEnforce.log.
       Find exact command line SCCM executed.
       Find exit code.

  2. Copy the same command and test manually as admin.

  3. Test as SYSTEM using PsExec:
       psexec -s -i cmd.exe
       Run install command from SYSTEM cmd.

  4. Add proper silent switch:
       MSI example:
         msiexec /i app.msi /qn /norestart

       EXE example:
         setup.exe /silent /norestart
         or vendor-specific silent parameter

  5. Set correct installation behavior:
       Install for system
       Whether or not user is logged on

  6. Add prerequisites as dependencies.

  7. Add expected success codes:
       0 = Success
       3010 = Success, reboot required

  8. Update content and redistribute after changing source files.

Verification:
  - AppEnforce.log shows exit code 0 or 3010.
  - AppDiscovery.log shows detection successful.
  - Software Center shows Installed.

11. APPLICATION INSTALLED BUT STILL SHOWS INSTALL BUTTON
============================================================

Symptoms:
  - Application installed successfully
  - Software Center still shows Install
  - Console shows Failed or Not installed
  - Reinstall happens repeatedly

Most common root causes:
  - Detection method is wrong
  - Wrong MSI product code
  - Wrong file path
  - 32-bit/64-bit registry path mismatch
  - App installs per-user but detection checks machine location
  - Detection checks old version

Important logs:
  AppDiscovery.log
  AppEnforce.log

Proven solution:
  1. Open AppDiscovery.log.

  2. Find detection result:
       App not discovered
       Detection failed
       Method returned false

  3. Fix detection method:
       For MSI:
         Use correct product code.

       For File:
         Correct path and file name.
         Example:
           C:\Program Files\Mozilla Firefox\firefox.exe

       For Registry:
         Check 32-bit vs 64-bit registry.
         Enable/disable "This registry key is associated with a 32-bit application on 64-bit systems" correctly.

  4. If app installs per-user:
       Use user-based detection path or registry under HKCU.
       Or install app machine-wide.

  5. Run Application Deployment Evaluation Cycle.

Verification:
  - AppDiscovery.log shows detection = true.
  - Software Center changes to Installed.

12. APPLICATION REQUIREMENT / DEPENDENCY PROBLEM
============================================================

Symptoms:
  - App not applicable
  - App not showing
  - App stuck waiting for dependency
  - Requirement not met

Most common root causes:
  - OS version requirement wrong
  - Architecture requirement wrong
  - Disk/RAM condition not met
  - Dependency app not deployed
  - Supersedence configured wrongly

Important logs:
  AppIntentEval.log
  AppDiscovery.log
  DCMAgent.log

Proven solution:
  1. Check AppIntentEval.log.
       This log tells why app is or is not applicable.

  2. Check application Deployment Type requirements.

  3. Remove unnecessary requirements for testing.

  4. Confirm dependency application:
       Has valid detection method
       Content is distributed
       Is deployed or set to auto install

  5. Check supersedence:
       Confirm old/new app relationship.
       Confirm uninstall/install behavior.

Verification:
  - AppIntentEval.log shows applicable.
  - Application appears or installs.

14. SOFTWARE UPDATES NOT SHOWING IN SOFTWARE CENTER
============================================================

Symptoms:
  - Update is deployed
  - Client requires update
  - Update not visible under Software Center > Updates

Most common root causes:
  - Deployment not targeted to client collection
  - Client not in Required status
  - Deployment hidden from Software Center
  - Available time not reached
  - Software Update Scan Cycle not completed
  - Deployment Evaluation Cycle not completed
  - Client setting disables software updates
  - SUP/WSUS scan failure

Important logs:
  UpdatesDeployment.log
  UpdatesStore.log
  WUAHandler.log
  ScanAgent.log
  PolicyAgent.log

Proven solution:
  1. Confirm client is in target collection.

  2. In All Software Updates, select update:
       Click View Required.
       Confirm target client is listed.

  3. Check deployment properties:
       Scheduling:
         Software available time = As soon as possible or past time.

       User Experience:
         Display in Software Center and show notifications.

  4. Check Client Settings:
       Enable software updates on clients = Yes.

  5. On client, run:
       Machine Policy Retrieval
       Software Updates Scan Cycle
       Software Updates Deployment Evaluation Cycle

  6. Check UpdatesDeployment.log:
       Confirm assignment is active and evaluated.

  7. Check UpdatesStore.log:
       Search KB number.
       Confirm Required/Missing state.

Verification:
  - Update appears under Software Center > Updates.
  - UpdatesDeployment.log shows actionable update.
  - UpdatesStore.log shows update is required.

15. SOFTWARE UPDATE SHOWS UNKNOWN STATUS
============================================================

Symptoms:
  - Console shows Unknown
  - Compliance report incomplete
  - Update status not changing

Most common root causes:
  - Client has not scanned against WSUS/SUP
  - WUA scan failed
  - Client cannot reach SUP
  - State message not sent
  - Client inactive
  - WSUS GPO conflict
  - SUP unhealthy

Important logs:
  WUAHandler.log
  ScanAgent.log
  UpdatesStore.log
  StateMessage.log
  CcmMessaging.log
  statesys.log on site server

Proven solution:
  1. On client, run Software Updates Scan Cycle.

  2. Check WUAHandler.log.
       Confirm scan started and completed.

  3. Check ScanAgent.log.
       Confirm SUP location is assigned.

  4. Check LocationServices.log.
       Confirm client finds SUP.

  5. Check Windows Update GPO:
       Domain GPO should not point clients to wrong WSUS.

  6. Check StateMessage.log.
       Confirm client sends state messages.

  7. On site server, check statesys.log.
       Confirm state messages are processed.

Verification:
  - Client moves from Unknown to Required/Compliant.
  - WUAHandler.log shows successful scan.

16. SOFTWARE UPDATE SCAN FAILED
============================================================

Symptoms:
  - Software update scan fails
  - WUAHandler.log shows error
  - Client remains Unknown
  - Updates never become Required

Most common root causes:
  - SUP/WSUS not reachable
  - Wrong WSUS URL in registry/GPO
  - WSUS IIS issue
  - Windows Update Agent problem
  - Proxy issue
  - Corrupt Windows Update cache

Important logs:
  WUAHandler.log
  ScanAgent.log
  LocationServices.log
  WindowsUpdate.log

Proven solution:
  1. Check LocationServices.log.
       Confirm SUP location is found.

  2. Check WUAHandler.log.
       Note exact error code.

  3. Confirm registry:
       HKLM\Software\Policies\Microsoft\Windows\WindowsUpdate
       WUServer should point to correct SUP/WSUS.

  4. If wrong GPO is applying, fix Group Policy.

  5. Reset Windows Update components if cache corruption suspected:
       net stop wuauserv
       net stop bits
       rename C:\Windows\SoftwareDistribution SoftwareDistribution.old
       net start bits
       net start wuauserv

  6. Run Software Updates Scan Cycle again.

Verification:
  - WUAHandler.log shows scan completed successfully.
  - UpdatesStore.log shows update status.

19. UPDATE DEPLOYMENT EXISTS BUT NOT ACTIVE
============================================================

Symptoms:
  - Update is deployed
  - Client receives assignment
  - Software Center does not show update
  - UpdatesDeployment.log says:
       Assignment is still not active
       Unable to evaluate assignment as it is not activated yet
       Total actionable updates = 0

Most common root causes:
  - Software available time is in the future
  - Client local time is wrong
  - Server time/time zone mismatch
  - Deployment uses UTC and time conversion is misunderstood

Important logs:
  UpdatesDeployment.log
  PolicyAgent.log

Proven solution:
  1. Open deployment properties.

  2. Go to Scheduling tab.

  3. Set:
       Software available time = As soon as possible

  4. For testing, set:
       Installation deadline = As soon as possible

  5. Check client date/time/time zone:
       date /t
       time /t
       tzutil /g

  6. Check SCCM server date/time/time zone.

  7. Run:
       Machine Policy Retrieval
       Software Updates Deployment Evaluation Cycle

Verification:
  - UpdatesDeployment.log no longer says "not active".
  - Update appears in Software Center.

21. AUTOMATIC DEPLOYMENT RULE FAILED
============================================================

Symptoms:
  - ADR does not create Software Update Group
  - ADR does not download updates
  - ADR does not deploy
  - Patch Tuesday automation fails

Most common root causes:
  - Search criteria returns no updates
  - Updates are expired/superseded
  - Deployment package path permission issue
  - Download failed
